1.3.x

1.3.3

Issue date: 04/18/2012

  • Performance improvement. Using new RapidBeans framework version 0.9.6 with tuned "hostname()" function.

1.3.2

Issue date: 04/01/2012

  • Changed installunit property removerootdirs to unpackstrategy. Please note that this change implies a small migration effort: removerootdirs="false" => unpackstrategy="simple" removerootdirs="true" or removerootdirs not defined usually => unpackstrategy="removeroot" Use unpackstrategy="removerootdeep" in case you are not sure.

  • Added text file encoding attribute for all kinds of file configurations (textfile, propertiesfile, xmlfile).

1.3.1

Issue date: 03/01/2012

  • Internal redesign of the concept of interpreted strings: Added the information which strings are really interpreted to the reference manual.

1.3.0

Issue date: 01/23/2012

  • Introduced install state data file ".renvstate.xml" in order to persist the installation state and to make install unit home directories independent from the name/version convention. Please note: this change requires migration of existing environment. Migration should happen automatically during the first "renv" call. If not simply install the installation unit again.

  • Introduced integration of ant tasks alternatively to system commands

  • Introduced input for commands with multiple lines. in order to allow automatic user interactions of e. g. installers.

  • Introduced message texts for commands

  • Improved HTTP download speed by factor 4

  • Improved local and share file copy speed for large files on Windows

1.2.x

1.2.0

Issue date: 11/03/2011

  • RapidEnv language changes (slight migration effort):

    • Renamed property attribute propmode to category

    • Renamed property attribute proptype to valuetype

    • Changed property category value individual to personal

  • Improved properties status display: show common properties { '=' , '!' } and personal properties 'p'.

  • Introduced explicit property modification: while "renv u <property-name> ..." will update one ore more common properties if necessary the same command given on a personal property will let you redefine its value. "renv c <property-name> ..." exactly behaves the same way.

  • Introduced folder configuration (in analogy to file configuration): the sourceurl property defines a packed file to unpack under the folder. If no packed file is specified the folder simply is created

  • Introduced element commandonconfig: for commands that should be executed whenever a file or folder has been (re) configured.

  • Introduced property deleteafterdeinstall: for configuration files (folders) outside the installation root (e. g. Eclipse links).

  • Replaced windows shortcut creation JShortcut solution with Windows scripting host solution.

  • Introduced replace() function expression for simple text replacement in expressions.

  • Started with Eclipse support: plugins, simple workspace configurations (build pat variables), projects, ...

1.1.x

1.1.0

Issue date: 08/01/2011

  • Introduced UNIX support with

    • a cmd line shell script wrapper renv (execute '. renv')

    • support of desktop icons maintenance (start menu support is still incomplete)

    • support of a "command prompt here" like Dolphin (KDE) and Nautilus (Gnome) menu extension

    tested und openSUSE Linux 11.4 together with KDE and Gnome

  • Extended unpacking of archive files in order to support not only zip format: Currently the following formats are supported: zip "*.zip jar "*.jar war "*.war tar "*.tar tar with gzip compression "*.tar.gz, *.tgz tar with b2zip compression "*.tar.bz2, *.bz2

  • Introduced expression function packaging() to determine platform specific packaging ("zip" vs. "tar.gz")

  • Introduced expression function userhome() to determine the user's home directory platform independently

  • Introduced conversionmap argument for expression functions osname(), architcture(), and packaging() in order to allow flexible determination of OS and architecture specific installation file names.

  • Fixed expression function homedir() which did not know its parent installunit in some cases.

  • Introduced attribute sourcefile to compensate complex download URLs.

  • Introduced icon configuration for shortcuts and start menu entries on the desktop (Windows and KDE)

  • Introduced special environment variable RAPID_ENV_PROJECT_HOME to determine the project development root directory. Defining a property mapped to this environment variable lets you define this project root commonly or individual. Once RAPID_ENV_PROJECT_HOME is defined renv will issue a warning if executed with a workind directory that is not below the project root. This will help developers not to get mixed up especially when working with multiple development environments for the same project e.g. when working with different branches.

  • Fixed deinstallation of old installation unit versions during up and downgrade. Upgrade and downgrade now perform a complete deinstallation instead of only removing the installation files. The version to deinstall will now be properly expanded into expressions used to define icons, deinstalltion commands etc...

  • Fixed exception for "renv -v" which is the shortcut for "renv -verbose stat"

  • Fixed several use cases dealing with path extensions including suppression of extendng path component '.' or './'.

  • Renamed install control literal deprecated to discontinued.

  • Fixed missing propagation of "discontinued" to sub installation units

  • Investigated reflexive command path definition When the command path is defined recursively during the next update all old values were contained including the path that should be removed. You have to use function "substractextions" around recursive value definitions and you have to mark path compomnents you want to remove with a property extision of kind "remove". Summary: do not define the command path as common property reflexively. Rather define an individual string property that is part of the common command path property.

  • Introduced new option "-yes": If you use this option the "boot", "update", or "configuration" will run without user interaction which causes individual properties to get the default values. Yes / No questions will be automatically answered with "Yes". So you should use this option for non-interactive (batch) build processes or scripts.

  • Adapted property update behaviour: Properties are from now on updated on demand (with commands "update" or "configure").

  • Introduced release notes
